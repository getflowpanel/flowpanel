Both implement the same `ResourceAdapter` contract defined in `@flowpanel/core`.

## Drizzle-specific notes

### Schema setup

`buildModelMetadata` requires the full schema object (tables **and** `relations()` exports)
to populate `relationModel` on relation fields.

```ts
// schema.ts
export const users = pgTable("users", { ... });
export const posts = pgTable("posts", { ... });
export const postsRelations = relations(posts, ({ one }) => ({
  author: one(users, { fields: [posts.authorId], references: [users.id] }),
}));

// flowpanel.config.ts
import * as schema from "./schema";
import { buildModelMetadata } from "@flowpanel/adapter-drizzle";

const postsMeta = buildModelMetadata(schema, "posts");
// postsMeta.fields includes { name: "author", kind: "relation", relationModel: "users", isList: false }
```

### Relational queries

The Drizzle adapter uses `db.query[model].findMany({ with: args.include })` for relation
eager-loading. This requires Drizzle's relational query API, which means:
- Pass the schema to `drizzle(db, { schema })` when creating the db instance.
- Declare all relations via `relations()` in the schema.

### Nested filter paths

Nested filter paths (e.g. `author.email`) are not translated to SQL `EXISTS` subqueries
in the current beta. The SQL filter builder skips them with a console warning.
Relational filtering at the application layer (post-fetch) is unaffected.
Full `EXISTS` subquery support is planned for v1.0.

## Prisma-specific notes

The Prisma adapter delegates directly to PrismaClient's fluent API. All features map
one-to-one. Relation includes are passed as Prisma `include` objects.
